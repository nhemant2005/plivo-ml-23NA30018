TL;DR

Five changes to the baseline survived isolation testing across eleven runs, and together they moved dev bpb from 2.3718 to 1.8798 (−20.7%). In order of contribution: batch 8→32 (−0.2529), learning rate 3e-4→1e-3 (−0.1065), a char-level tokenizer with byte fallback at vocab 913 (−0.0646), width n_embd 160→184 (−0.0488), and gradient clipping at max_norm=1.0 (−0.0192). Everything else regressed and was reverted.

The unifying reason is that a 2,000-step budget leaves this model data-starved and under-trained, not capacity-limited. That single fact explains both the wins and the losses: batch size and learning rate win biggest because both buy more learning per step; the tokenizer wins because 1.283 bytes/token cuts the number of NLL terms that bpb divides by; and LR decay, warmup, GPT-2 scaled init, and weight decay all lose because each trades early progress for a late-training benefit the run never survives long enough to collect.

The most instructive result was the last one. The learning rate was inherited from the baseline and never varied for nine runs. Phase 1's three "LR schedule" runs had held peak LR equal to the baseline's constant, so they unintentionally tested LR magnitude rather than schedule shape — and sorted by mean LR they form a clean monotonic curve whose best point is the highest LR ever tried. Acting on that reading, one run at lr=1e-3 delivered −0.1065: more than the tokenizer and width changes combined, and enough to falsify Phase 1's conclusion that the baseline was near its capacity limit rather than paying a mediocre-trainer tax. It neither diverged nor turned the loss curve over, so the optimum is bracketed only from below and is likely higher still.

Known-open levers (measured or bracketed, not closed)

Learning rate — identified by audit, tested in Run 10, still open above 1e-3. Runs 0–9 all used lr=3e-4, inherited unchanged from the baseline. The Phase 1 schedule runs held peak LR equal to that constant, so all three ran at roughly half the average LR — they varied LR magnitude, not schedule shape. Sorted by mean LR they are cleanly monotonic (1.50e-4 → 2.6976, 1.63e-4 → 2.6338, 1.65e-4 → 2.6076, 3.00e-4 → 2.3718), best result at the highest LR ever tested. Run 10 acted on this: lr=1e-3 gave 1.8798 (−0.1065), the second-largest win of the exercise. It did not diverge, and while the late slope flattened (−0.0288/250 steps vs Run 9's −0.0518) the curve still has not turned over — the optimum is bracketed only from below (3e-4 < 1e-3 ≤ ?). 2e-3, and a warmup+cosine schedule with peak above 1e-3 — the hypothesis Phase 1 never actually tested — are the obvious next runs. Out of time, not out of headroom.

block_size 128 → 256. Rejected on wall-clock cost; its benefit was never measured. It is cheaper than it looks — solving the parameter formula at vocab=913 gives 48e² + 2136e = 2e6 → n_embd=180, i.e. 4 units of width for 2× the context, with eval context doubling too since evaluate.py strides at block_size // 2.

Seed variance. Never measured; every run used seed=1337. The large wins (batch −0.2529, LR −0.1065, tokenizer −0.0646, width −0.0488) are comfortably outside plausible noise; the small deltas (grad clip −0.0192, tying +0.024, AdamW +0.0071) are not, and those three verdicts should be read as provisional.

BPE. The char tokenizer reaches 1.283 bytes/token. BPE at ~2.5–3 bytes/token would be roughly twice the leverage of the byte→char step, and is the point at which weight tying would finally pay (at vocab 4096 tying frees ~750K params, against 146K at vocab 913). Closed for implementation risk and time, not for measured evidence.

Best configuration & why (NOTES.md, verbatim)

Best configuration: char-level tokenizer with byte fallback (vocab 913) + 4 layers / 4 heads / n_embd=184 / block=128, untied embeddings, plain Adam at lr=1e-3, gradient clipping at max_norm=1.0, batch=32, 2000 steps — dev bpb 1.8798, 1,994,560 params (99.7% of the 2M cap), down from the 2.3718 baseline (−20.7%).

Five changes earned their place, each isolated against the recipe below it: batch 8→32 (−0.2529), learning rate 3e-4→1e-3 (−0.1065), the char+byte tokenizer (−0.0646), width n_embd 160→184 (−0.0488), and gradient clipping (−0.0192). Everything else regressed and was reverted. The tokenizer wins because the corpus uses only 657 distinct codepoints while Devanagari costs 3 bytes/char — dropping to 1.283 bytes/token directly cuts the number of NLL terms that bpb divides by. Width was sized by solving the exact parameter formula 4·(12e²+13e) + 2·913·e + 128·e + 2e against the cap rather than by trial and error.

The recurring lesson across ten runs is that a 2,000-step budget punishes anything that trades early progress for late refinement: LR decay, warmup, and GPT-2 scaled init all regressed for the same reason, and weight decay had nothing to regularise because the model never reaches the overfitting regime. Every checkpoint, including the best one, is still descending steeply at step 2000 (slope −0.05 per 250 steps and not flattening), so this model is under-trained rather than at its capacity limit — which is why the largest remaining lever is almost certainly the learning rate, the one variable never varied across any run.

Tokenizer correctness

Losslessness (decode(encode(text)) == text) is a hard cap, and is re-verified inside evaluate.py on every scoring call. It was independently checked before training on: the full 7.3MB train_corpus.txt; dev_eval.txt; a 2000-char Devanagari-only slice (multi-byte UTF-8 boundaries are where a naive char/merge tokenizer silently corrupts); and an out-of-vocab stress string of emoji, Cyrillic, Chinese and accented Latin, none of which appear in the 657-char vocab. All exact.

The byte-fallback path is not dead code on real data: dev_eval.txt contains 9 characters (ⳉ ⋅ پ ش ⲃ ⲱ ⲧ ف Ø) absent from the training vocab, which encode to 2–3 fallback byte ids each and round-trip exactly.

Full run log (RUNLOG.md, verbatim)

Hypothesis, change, result and conclusion for every run, including the six that regressed and were reverted. 

RUN 0 — baseline
  Hypothesis: n/a, establish baseline.
  Config: constant lr=3e-4, Adam, batch=8, block=128, 4L/4H/160d, no schedule/decay/clip, byte tokenizer.
  Result: train loss 5.65 -> 1.73, dev bpb 2.3718, params 1,339,840.
  Conclusion: baseline works, matches handout description of "mediocre on purpose."

RUN 1 — LR warmup + cosine decay to 0
  Hypothesis: warmup + cosine decay reduces late-training noise and should beat constant LR.
  Change: 100-step linear warmup then cosine decay to 0 over remaining 1900 steps.
  Result: train loss plateaued ~1.96-1.97 (vs baseline 1.73). Dev bpb 2.6976.
  Conclusion: regression. Suspected decay-to-zero throttles the back half of a 2000-step run.

RUN 1b — same schedule, decay to 10% floor instead of 0
  Hypothesis: a floor lets the model keep learning late instead of freezing.
  Result: train loss ~1.93. Dev bpb 2.6338 — still worse than baseline.
  Conclusion: floor alone didn't fix it. Step-100 loss: baseline 2.85 vs this run 3.63 — the warmup
  ramp is costing early steps a 2000-step budget can't recover. Next: isolate warmup by removing it.

RUN 1c — decay to 10% floor, no warmup
  Hypothesis: warmup was the culprit; decay-only should recover baseline performance.
  Result: step-100 loss matches baseline (2.85), step-2000 ~1.90. Dev bpb 2.6076 — still worse.
  Conclusion: it's the decay itself, not warmup. Early training now matches baseline exactly, but
  decaying LR while the model is still improving fast at 3e-4 constant costs more than the noise
  reduction earns. Reverted; schedule code removed rather than left as dead machinery.

RUN 2 — weight tying (vocab 256)
  Hypothesis: frees ~41K params and often improves perplexity via shared input/output embedding.
  Result: params 1,339,840 -> 1,298,880. Dev bpb 2.4122 (+0.0404).
  Conclusion: mild regression standalone. At n_embd=160 the two roles (token lookup vs next-token
  prediction) appear to want different directions. Re-evaluate once vocab is larger.

RUN 3 — scaled init + gradient clipping (combined)
  Hypothesis: GPT-2 init (std=0.02, residual branches scaled by 1/sqrt(2*n_layer)) plus clipping
  should be neutral-to-positive.
  Result: step-100 loss 3.01 (vs baseline 2.85). Dev bpb 2.4465.
  Conclusion: regression. Slower early loss points at the smaller base std. Need to isolate.

RUN 3b — gradient clipping alone (init reverted to std=0.05)
  Result: step-100 loss 2.7734 (better than baseline 2.8522), step-2000 1.7133. Dev bpb 2.3526.
  Conclusion: WIN, confirmed — first improvement over baseline (-0.0192). Better throughout the
  whole curve, not just the tail: cleaner gradient steps from early on. Confirms Run 3's regression
  traces to the init, not the clip.

RUN 3c — scaled init alone
  Result: step-100 loss 3.0864 (worse than baseline and worse than combined Run 3). Dev bpb 2.4519.
  Conclusion: standalone regression, culprit confirmed. At 4 layers the residual-variance problem
  this guards against barely exists, while std=0.02 genuinely slows early convergence. GPT-2 init
  tricks are tuned for much deeper nets.

RUN 4 — AdamW + weight_decay=0.1 (stacked on grad clip)
  Result: step-2000 loss 1.7219 (vs 1.7133). Dev bpb 2.3597 (+0.0071).
  Conclusion: small regression, reverted. At 2000 steps the model never reaches the overfitting
  regime, so decay just costs capacity. Phase 1 verdict: 1 of 5 trainer changes helped.

RUN 5 — batch size 8 -> 32 (byte tokenizer, isolated)
  Hypothesis: batch cost is ~linear while block cost is quadratic; batch=32 sees 8.19M tokens
  (>1 epoch of the 7.3M-byte corpus) and reduces per-step gradient noise.
  Result: step-2000 loss 1.4254 (vs 1.7133). Dev bpb 2.0836 (-0.2690).
  Conclusion: WIN by a wide margin — batch alone beats every Phase 1 trainer change combined.
  Banked; reverted to batch=8 for tokenizer development so iteration stays fast (7.4min vs 1.9min).

RUN 6 — char-level + byte-fallback tokenizer (batch=8, isolated)
  Hypothesis: corpus has only 657 unique codepoints; char-level removes most of the byte-encoding
  tax (28% sequence inflation; Devanagari costs 3 bytes/char) at far lower risk than merge-trained BPE.
  Losslessness verified before training: full corpus, dev_eval, a 2000-char Devanagari slice, and an
  OOV stress string (emoji/Cyrillic/Chinese/accented Latin) — all exact round-trips.
  Result: params 1,550,080 (78% of cap). Dev bpb 2.2880 (-0.0646).
  Conclusion: WIN, confirmed. Smaller than the -0.15..-0.30 estimate but real and in the predicted direction.

RUN 7 — weight tying retest at vocab=913
  Hypothesis: Run 2 was measured at vocab=256 where tying freed only ~41K; at vocab=913 it frees ~146K.
  Result: params 1,550,080 -> 1,404,000. Dev bpb 2.3120 (+0.024) — same shape as Run 2.
  Conclusion: regression, reverted. The embedding roles still want to diverge regardless of vocab size.
  Closed for both vocab sizes tried.

RUN 8 — width increase, n_embd 160 -> 184 (isolated)
  Hypothesis: Run 6 used only 78% of the cap. Solving total(V,e,L) = L*(12e^2+13e) + 2Ve + block*e + 2e
  for V=913, L=4, block=128 gives the largest n_embd divisible by n_head=4 as 184 -> 1,994,560 params.
  Result: params 1,994,560 (99.7% of cap). Step-2000 loss 2.0735 (vs 2.1022). Dev bpb 2.2392 (-0.0488).
  Conclusion: WIN, confirmed. Converged cleanly with the added capacity; no under-fitting.

RUN 9 — batch 8 -> 32, stacked on Run 8's recipe
  Hypothesis: Run 5's batch win (byte tokenizer) should stack rather than cancel on top of Run 8's
  tokenizer + width, since data coverage per step and model capacity are largely independent levers.
  Result: step-2000 loss 1.7102 (vs 2.0735). Dev bpb 1.9863 (-0.2529).
  Conclusion: WIN, confirmed, stacks cleanly — -0.2529 stacked vs -0.2690 for the same lever isolated
  in Run 5, i.e. ~94% of the isolated effect survives stacking.

RUN 10 — learning rate 3e-4 -> 1e-3 (batch=32, stacked on Run 9)
  Hypothesis: an audit of Runs 0-9 found lr=3e-4 was inherited from the baseline and NEVER VARIED in
  any run. Phase 1's three "LR schedule" runs all held peak LR equal to the baseline's constant, so
  each ran at roughly HALF the baseline's average LR — they tested LR magnitude, not schedule shape.
  Sorted by mean LR, all four Phase 1 points are cleanly monotonic (1.50e-4 -> 2.6976, 1.63e-4 ->
  2.6338, 1.65e-4 -> 2.6076, 3.00e-4 -> 2.3718): more LR, better bpb, every time, with the best result
  at the HIGHEST LR ever tested — the edge of the search range, not an interior optimum. Every
  checkpoint including Run 9 is also still descending steeply at step 2000 (slope -0.05/250 steps),
  i.e. under-trained rather than capacity-limited. Batch=32 additionally lowers gradient-noise scale,
  which raises the maximum stable LR. Predicted: a higher constant LR should be a large win.
  Change: --lr 1e-3 (~3.3x). Everything else identical to Run 9 — isolating LR alone. 1e-3 chosen over
  2e-3 as the expected-value play given only one run fit in the remaining time; grad clip already
  active as divergence insurance.
  Result: step-100 loss 3.0364 (worse early, as expected at higher LR), step-2000 loss 1.5502 (vs
  Run 9's 1.7102). Dev bpb 1.8798 (-0.1065).
  Conclusion: WIN — the second-largest single win of the exercise, larger than the tokenizer (-0.0646)
  and width (-0.0488) changes COMBINED. Phase 1's conclusion that "most of the baseline's 2.37 is
  closer to real capacity limits than mediocre trainer tax" is falsified: one untested hyperparameter
  was worth more than all of Phase 2b and Phase 3. No divergence at 3.3x LR; late slope flattened
  (-0.0288/250 steps vs Run 9's -0.0518) but the curve still has not turned over, so the optimum is
  likely higher still. Shipped as ckpt.pt.
  Open, out of time: LR bracketed only from below (3e-4 < 1e-3 <= ?). 2e-3 and warmup+cosine with peak
  >1e-3 — the hypothesis Phase 1 never actually tested — are the obvious next runs. Also untested:
  block_size 128->256 (costs only n_embd 184->180 by the param formula) and seed variance.
